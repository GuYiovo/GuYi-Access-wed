API 概览与鉴权

GuYi Aegis 提供极简的单端点 API 设计。所有操作均通过向同一 URL 发送 POST 请求，并通过参数区分行为。

POST https://your-domain.com/Verifyfile/api.php
# Header 支持: application/json (推荐) 或 x-www-form-urlencoded

关于 AppKey 鉴权机制

每一个请求都必须携带 app_key 参数。该参数是在管理后台创建应用时自动生成的 64 位十六进制字符串。它不仅用于区分应用，**更是 AES-256-GCM 响应加密的 32 字节核心密钥**。

安全告警：千万不要将 AppKey 明文写在代码中（尤其是易语言、C# 等易被反编译的语言）。请务必使用 VMP、混淆器加固，或通过动态算法在内存中拼接。

卡密验证接口

此接口负责：验证有效性、绑定设备、更新到期时间。如果是首次使用的卡密，系统会自动将其激活并绑定传入的硬件指纹。

请求参数 (Request Body)

字段名类型约束说明
app_keyString必传64位十六进制应用密钥
card_codeString必传用户输入的卡密，支持传 card
device_hashString可选设备唯一硬件特征的 MD5。若不传则降级使用客户端 IP。

响应格式 (成功/200 OK)

若服务器启用加密且 AppKey 合法，返回的 JSON 将是 {"encrypted_data": "BASE64..."}，解密后的真实 JSON 结构如下：

{
  "code": 200,
  "msg": "首次激活成功",
  "data": {
    "expire_time": "2025-12-31 23:59:59",
    "variables": {
      "update_url": "https://..."
    }
  }
}

错误状态字典

  • 400 - 参数缺失（如未传卡密）。
  • 403 - 验证失败（包含：卡密过期、已被封禁、绑定了其他设备、AppKey 错误等，具体看 msg 字段）。
  • 429 - 频率超限（详见下文频率限制）。

AES-256-GCM 解密规则

为了防止中间人篡改（例如修改 expire_time 为 2099 年），我们的加密采用了带认证标签的 GCM 模式。

  • 密文编码：Base64
  • 二进制结构：IV (12字节) + Auth Tag (16字节) + Ciphertext (剩余字节)
  • 解密密钥：将 64 位的 app_key 字符串直接进行 Hex 解码（Hex to Bytes），得到 32 字节的密钥。

👉 请在右侧（或下方）面板中选择您的编程语言，获取完整的请求与解密源码。

心跳保活与离线容错 (企业级建议)

1. 频率限制保护

系统底层实施严格的并发控制，单 IP 每分钟最多允许验证 60 次。触发限制后将返回 429。因此，严禁客户端每秒高频调用验证接口。

2. 心跳设计 (最佳实践)

建议客户端在首次验证成功后，将 expire_time 存入本地内存。在软件运行期间，每隔 30 分钟 ~ 1 小时 向服务器发起一次静默验证即可。

3. 离线容错机制

当用户的网络波动导致 API 请求超时或失败时（如捕获到 SocketException），建议允许用户在 2~6 小时的宽限期内 依靠本地缓存的 expire_time 继续使用软件，提升用户体验。
